### How to Update Documentation

**For Feature Changes:**
1. Update [planning/ROADMAP.md](planning/ROADMAP.md) if user-facing
2. Update [planning/DEVELOPMENT_ROADMAP.md](planning/DEVELOPMENT_ROADMAP.md) if technical

**For Bug Fixes:**
1. Update [planning/TASKS.md](planning/TASKS.md) status
2. Document in [implementation/POTENTIAL_ISSUES.md](implementation/POTENTIAL_ISSUES.md) if recurring

**For New Features:**
1. Add to [planning/ROADMAP.md](planning/ROADMAP.md)
2. Create task in [planning/TASKS.md](planning/TASKS.md)
3. Add specs to [planning/SPECIFICATIONS.md](planning/SPECIFICATIONS.md)
4. Document implementation in [implementation/](implementation/)

### Review Schedule
- **Weekly:** Check feature verification accuracy
- **Monthly:** Update roadmaps and task status
- **Quarterly:** Full documentation audit
- **Per Release:** Update all user-facing docs

## ðŸ“ Contributing to Documentation

We welcome documentation improvements! When contributing:

### Style Guidelines
- Use Markdown format
- Include table of contents for docs >200 lines
- Use emojis for visual navigation (sparingly)
- Keep language clear and concise
- Include code examples where relevant

### File Naming
- Use SCREAMING_SNAKE_CASE for guides (e.g., `BUILDING.md`)
- Use descriptive names that match content
- Prefix with category if in root (`FEATURE_VERIFICATION_REPORT.md`)

### Organization Principles
- **planning/** = Future-focused (what we're building)
- **implementation/** = Present-focused (what we've built)
- **development/** = Process-focused (how we build)
- **archive/** = Historical (what we built when)
